# Extending the JVM backend (implementing abstract features)

Read **eliot-layers** first — it owns *placement* (which layer/pool a name goes in, abstract↔concrete merge,
the two-pool model). This skill is the *backend* half: once you know a name needs a **runtime realization**, how
do you actually make bytecode come out — and what compile-time counterpart the checker needs alongside it.

## The one-paragraph model

The base stdlib declares things **abstractly**: a body-less `def` (`printLine`, `stringEquals`, the `+` width
dispatch's `nativeAdd…` leaves), an abstract `type` (`Bool`, `String`, `IO`), an `ability` with no instance for a
platform type (`Eq[String]`). "Implement it on the JVM" = redefine the same name in the **`jvm` layer** with a body
/ `data` / instance, and — for the handful of names with no Eliot body at all — supply the **native leaf** as raw
bytecode in the backend. A name that the *compiler itself must evaluate* at check time additionally needs a
**compile-time reduction** (a Scala `VNative` in a `*NativesProcessor`). These two realizations are independent
mechanisms keyed by the same FQN and never conflict (see [[project_compiler_as_platform]]).

Most "implementations" are ordinary Eliot in the jvm layer (`.els`, an `implement`, a `data` body) — that is
eliot-layers, no backend code. You only touch **Scala backend files** when the leaf bottoms out at bytecode no
Eliot body can express: an I/O syscall, a machine op, a boxed-primitive comparison, a branch.

## How a name reaches bytecode — the dispatch

`JvmClassGenerator.createModuleMethod` + `ExpressionCodeGenerator` route every used value one of four ways. Know
which one you are adding:

| Path | For | Where you register it | Emission |
|---|---|---|---|
| **Eliot body** | anything expressible in Eliot | just the `.els` (jvm layer) | a generated static method from the body — no backend code |
| **native method** | a **fixed-signature, non-generic** leaf (`printLineInternal`, `readLineInternal`, `unit`, `stringEquals`) | `NativeImplementation.implementations` map | one generated static method with a hand-written body |
| **intrinsic (inline)** | **generic / control-flow / nullary-constant / per-instantiation** leaf (`nativeAdd…`, `nativeWiden`, `intToString`, `true`/`false`/`fold`/`&&`) | `Intrinsics.all` set + a case in `ExpressionCodeGenerator.generateIntrinsic` | bytecode spliced **at each call site**, no method |
| `data` ctor / `handleCases` / `typeMatch` | generated by `DataClassGenerator` | (automatic from `data`) | — |

Anything else body-less → **`"Function not implemented."`** abort. So the two backend paths you author are
**native method** and **intrinsic**, and the decision is mechanical:

> **Native method** if the leaf is non-generic with one fixed JVM descriptor (its call site resolves to exactly one
> method name). **Intrinsic** if it is generic (its descriptor varies per instantiation — a single static method
> can't serve `nativeAdd[Byte]` and `nativeAdd[Long]`), takes/returns *nothing* (`true`/`false`), or **is control
> flow** (`fold` must branch, not push three operands). When unsure, native method is simpler; reach for intrinsic
> only when the descriptor varies or you need a jump.

Why non-generic ⇒ native method works: the call site invokes `mangledMethodName(fqn, typeArgs)`
(`CommonPatterns`), which for **empty typeArgs + `Qualifier.Default`** is the **bare name** — matching the fixed
`JvmIdentifier("name")` the native method is generated under. A generic leaf gets a `$TypeArg` suffix that no
single method can match, which is exactly why generics go inline.

## Backend file inventory (all under `jvm/src/.../jvm/classgen/`)

- `asm/NativeType.scala` — the FQN → `java.lang.*` map (`types`) and the descriptor helpers
  (`javaSignatureName`/`javaInternalName`/`convertToSignatureString`). **Every abstract type that appears in a
  runtime signature must be in `types`** or it gets a bogus nested-class descriptor (`eliot/lang/Bool$Bool`) that
  doesn't exist. Keys use `systemLangType(name)` = `Qualifier.Default` (type FQNs are stripped to Default by the
  time they reach here; `WellKnownTypes` spells them `Qualifier.Type`).
- `processor/Intrinsics.scala` — the `all: Set[ValueFQN]` of inline-emitted FQNs. Adding an FQN here does **two**
  things: routes it to `generateIntrinsic`, and excludes it from the body-less "not implemented" abort
  (`JvmClassGenerator` filters `!Intrinsics.isIntrinsic(k)`).
- `processor/ExpressionCodeGenerator.scala` — `generateIntrinsic` (author your inline case here), and the call
  dispatch (`generateFunctionApplication` → intrinsic vs `generateNormalFunctionCall`). `createExpressionCode`
  emits any sub-expression; interleave it with `methodGenerator.runNative { mv => … }` for raw ASM.
- `processor/NativeImplementation.scala` — the `implementations: Map[ValueFQN, NativeImplementation]`. Each entry
  hand-writes a method via `classGenerator.createMethod(JvmIdentifier(name), paramTypeFQNs, returnTypeFQN).use { mg
  => mg.runNative { mv => … } }` (**`createMethod` appends the `*RETURN`** — leave the result on the stack). Set
  `override val impure = true` for a leaf that touches the world (I/O) or diverges (`forever`); the backend then
  **asserts the def is declared `private`** (`verifyNativeVisibility`). Pure leaves stay `impure = false`.
- `processor/JvmClassGenerator.scala` — the dispatch above (`createModuleMethod`). Read-only for most tasks; the
  leaf is emitted into **the class named by its FQN's module** and only if `used` reached it.
- `asm/CommonPatterns.scala` — `mangledMethodName`, `valueType` (a `GroundValue`'s erased carrier FQN).

## Recipe A — a native-method leaf (fixed signature)

Worked example: `stringEquals(a: String, b: String): Bool` behind `Eq[String]`.

1. **Declare** the body-less `private def stringEquals(a: String, b: String): Bool` in the jvm `.els` that hosts
   the instance (eliot-layers decides the file). `private` iff impure — here pure, but `private` keeps user code on
   the public op.
2. **Map any abstract type** in the signature: `Bool` isn't a `data`, so add
   `(systemLangType("Bool"), eliot_lang_Bool)` → `classOf[java.lang.Boolean]` in `NativeType.types`.
3. **Register the method** in `NativeImplementation.implementations`:
   `(systemLangValueFQN("Eq", "stringEquals"), eliot_lang_Eq_stringEquals)`, and write the body — push args, call,
   leave the result:
   ```scala
   mv.visitVarInsn(Opcodes.ALOAD, 0); mv.visitVarInsn(Opcodes.ALOAD, 1)
   mv.visitMethodInsn(Opcodes.INVOKEVIRTUAL, "java/lang/String", "equals", "(Ljava/lang/Object;)Z", false)
   mv.visitMethodInsn(Opcodes.INVOKESTATIC, "java/lang/Boolean", "valueOf", "(Z)Ljava/lang/Boolean;", false)
   ```
4. **Compile-time counterpart** if the checker must evaluate it (see below).

## Recipe B — an inline intrinsic (generic / constant / control flow)

Worked example: the `Bool` ops. Represent `Bool` as `java.lang.Boolean` (Recipe A step 2), then:

1. Add the FQNs to `Intrinsics` (a `langBool(name)` helper mirroring `langInt`) and into `all`.
2. Add a case in `generateIntrinsic`. Constants and stack ops are trivial; **control flow emits only the taken
   branch** — for `fold(cond, whenTrue, whenFalse)`:
   ```scala
   val elseL = new Label(); val endL = new Label()
   createExpressionCode(…, arguments(0)) >>                                   // cond → Boolean
   runNative { mv => unboxBool(mv); mv.visitJumpInsn(IFEQ, elseL) } >>
   createExpressionCode(…, arguments(1)) >>                                   // whenTrue arm only
   runNative { mv => mv.visitJumpInsn(GOTO, endL); mv.visitLabel(elseL) } >>
   createExpressionCode(…, arguments(2)) >>                                   // whenFalse arm only
   runNative { _.visitLabel(endL) } >>
   methodGenerator.addCastTo(valueType(expectedResultType))                   // unify the merge type
   ```
   Share `Label`s across the `runNative` blocks; `COMPUTE_FRAMES` computes the stack-map at the merge (both arms
   must leave one value). Emitting arms in the branches is legitimate: Eliot is pure, so a skipped arm has no
   observable effect (and effectful arms are suspended `IO` values anyway).
3. No `NativeImplementation` entry, no descriptor — it never becomes a method.

## The compile-time counterpart (`*NativesProcessor`)

A runtime leaf that the **checker must evaluate at compile time** (a type-level use, a `where` guard bound, an
`auto`/calculated position, or self-sufficiency for the LSP where jvm is absent) needs a **compile-time reduction**
— a Scala `VNative` rule keyed by the same FQN. Two homes:

- `lang/.../monomorphize/processor/SystemNativesProcessor.scala` — compiler intrinsics the compiler *reasons
  about*: `Function`, `Type`, `true`/`false`/`fold`, `typeEquals`, `integerLiteral`.
- `stdlib/.../plugin/StdlibNativesProcessor.scala` — library ops the compiler *supplies but doesn't reason about*:
  the `BigInteger` arithmetic backing `Int` bounds, `&&`/`||`/`!`, `stringEquals`. Add here by default.

Shape: a curried `VNative` that reduces **only when its arguments are concrete `VConst`**, else returns
`stuck(fqn, args*)` (a `VStuckNative`) so the unifier falls back and `Evaluator.renormalize` re-fires it once the
args concretise. Fail-safe by construction — a leaf reaching the checker with non-concrete args stalls loudly, it
never answers wrong.

The two realizations coexist: `StdlibNativesProcessor.stringEquals` (checking) + `NativeImplementation`
`stringEquals` (codegen) are different mechanisms on the same FQN — the merger reads the native for checking, the
backend emits the leaf for codegen, no conflict. Register a new stdlib label with the merger via the plugin's
`configure()`; the two existing labels are already wired.

**Ability instances across tracks.** An `implement X[T]` needs to *resolve* on whichever pool asks. A jvm runtime
instance is *borrowed* into the compiler pool for resolution, so often no compiler copy is needed. Add a
`stdlib/eliot-compiler/` overlay copy only when the compiler must be **self-sufficient** (LSP type-checks without
jvm) or must *evaluate* the instance's leaf at compile time — mirror `Eq[String]` (jvm instance + eliot-compiler
overlay + `StdlibNativesProcessor` reduction) and the `Either`/`Option` carriers. Compiler-*intrinsic* instances
whose leaf has no runtime meaning (types are erased) live in `lang` on both pools — that is `Eq[Type]`.

## Why the runtime was empty — the folding reality (read before "it doesn't run")

The single NbE evaluator forces every **pure** computation at compile time; the monomorphized body that reaches
codegen is the **normal form** (`PostDrainQuoter` reads it back structurally). So a lower-layer pure op **cannot be
observed at runtime unless one operand is a *stuck neutral*** — something the evaluator can't reduce, which bottoms
out at an **effect result** (a native leaf like `readLine`) or a runtime function parameter. This is why `Bool`
had no runtime bytecode for years: every `Bool` traced back to `true`/`false` literals and folded away. **To
exercise your leaf at runtime, drive it from an effect** (`readLine == "yes"`), not from constants.

`PostDrainQuoter` specifics worth knowing:
- Value refs and applications are read back **structurally** — `&&(a,b)` survives to codegen as a
  `MonomorphicValueReference` application. That is *how* your intrinsic case gets reached.
- `fold` is special (`trySelectFold`): it evaluates **only the condition**; a ground `true`/`false` collapses to
  the chosen arm at compile time, an abstract condition leaves an ordinary `fold` application for the backend.
- `trySelectFold` does **not** resolve ability calls, so a *constant* `"a" == "a"` (which reaches its leaf through
  the `Eq` ability) does **not** compile-time-fold in a fold condition — it runs. Direct natives (`&&`) do fold.
  Not a bug; the reduction still fires in resolved-ability positions (guards), same as `typeEquals`.
- A fully-ground compile-time value is *materialised* back to a literal / constructor / `true`/`false` ref
  (`materialise`), which is why a folded `Bool` still needs `true`/`false` handled at codegen.

## Verify

1. `./mill jvm.compile stdlib.compile` — fast Scala check of the backend edits.
2. Build + run a program that **drives the leaf from an effect** so it isn't folded:
   `./mill examples.run jvm exe-jar examples/src/ -m <M>` then `echo <input> | java -jar target/<M>.jar`.
3. **Confirm it's actually there**: `javap -c -p -classpath target/<M>.jar <Class>` — look for your `invokestatic`
   / `GETSTATIC` / `IFEQ`, not a folded constant. `unzip -l target/<M>.jar` shows which classes survived.
4. `./mill jvm.test lang.test stdlib.test` — the jvm integration tests run the whole pipeline; also run an
   Int-arithmetic example (`Ranges`, `ImplicitIntReturn`) since arithmetic shares the intrinsic/`NativeType` path.

## Gotchas

- **Abstract type absent from `NativeType.types`** → bad descriptor / `NoClassDefFoundError`. Map it (to a boxed
  wrapper, never a primitive — carriers are `Object`-erased).
- **Generic leaf as a native method** silently mismatches the mangled call name → "not implemented" or a link
  error. Generics go inline.
- **Forgot `Intrinsics.all`** → your inline FQN hits the body-less abort instead of `generateIntrinsic`.
- **Impure leaf not `private`** → build-time `visibilityViolation`. Set `impure = true` *and* declare it `private`.
- **`createMethod` already appends the return** — leave exactly one value (or none, for `Unit`→`Void`/`ACONST_NULL`)
  on the stack; don't emit your own `RETURN`.
- **FQN qualifier**: `systemLangType`/`langInt`/`langBool` build `Qualifier.Default` FQNs (what reaches the backend
  and `NativeType`); `WellKnownTypes` type FQNs are `Qualifier.Type`. Match Default for backend maps.
- **`.els` imports**: only `Function/Unit/String/BigInteger/IO/Int/Runtime` + `Type` are ambient; `Bool`, `Eq`,
  every effect need an explicit `import`. A missing import shows as "Name not defined" (with a cascade onto callers).
- **A leaf is only emitted if `used` reaches it** — an unreferenced native produces no method; test through a real
  call from `main`.
