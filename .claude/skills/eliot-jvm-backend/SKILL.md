# Extending the JVM backend (implementing abstract features)

Read **eliot-layers** first — it owns *placement* (which layer/pool a name goes in, abstract↔concrete merge,
the two-pool model). This skill is the *backend* half: once you know a name needs a **runtime realization**, how
do you actually make bytecode come out — and what compile-time counterpart the checker needs alongside it.

## The one-paragraph model

The base stdlib declares things **abstractly**: a body-less `def` (`printLine`), an abstract `type` (`Bool`, `String`,
`IO`), a body-less ability *instance* whose methods are native leaves (`implement Eq[String] { def equals }`, and the
`+`/`-`/`*` width-dispatch leaves are the `implement Numeric[Int]` methods `add`/`subtract`/`multiply`). "Implement it on the JVM" = redefine the same name in the **`jvm` layer** with
a body / `data` / instance, and — for the handful of names with no Eliot body at all — supply the **native leaf** as raw
bytecode in the backend (either an exact-FQN entry, or, for a body-less impl method, a native attached directly to it). A name that the *compiler itself must evaluate* at check time additionally needs a
**compile-time reduction** (a Scala `VNative` in a `*NativesProcessor`). These two realizations are independent
mechanisms keyed by the same FQN and never conflict (see [[project_compiler_as_platform]]).

Most "implementations" are ordinary Eliot in the jvm layer (`.els`, an `implement`, a `data` body) — that is
eliot-layers, no backend code. You only touch **Scala backend files** when the leaf bottoms out at bytecode no
Eliot body can express: an I/O syscall, a machine op, a boxed-primitive comparison, a branch.

## How a name reaches bytecode — the dispatch

`JvmClassGenerator.createModuleMethod` + `ExpressionCodeGenerator` route every used value one of four ways. Know
which one you are adding:

| Path | For | Where you register it | Emission |
|---|---|---|---|
| **Eliot body** | anything expressible in Eliot | just the `.els` (jvm layer) | a generated static method from the body — no backend code |
| **native method** | a **fixed-signature, non-generic** leaf keyed by exact FQN (`printLineInternal`, `readLineInternal`, `unit`) | `NativeImplementation.implementations` map | one generated static method under the leaf's own name |
| **native on an ability-impl method** | a body-less `implement X[T] { def m }` whose `m` *is* a backend leaf (`Eq[String]::equals` → `String.equals`) | `NativeImplementation.abilityImplementations` table (marker-recognized in `JvmClassGenerator`) | one static method under the impl method's **mangled** name |
| **intrinsic (inline)** | **generic / control-flow / nullary-constant / per-instantiation** leaf (the `Numeric[Int]` methods `add`/`subtract`/`multiply`, `Compare[Int]`'s `lessThanOrEqual`, `Show[Int]`'s `show`, `true`/`false`/`fold`/`&&`) | `Intrinsics.all` set (exact FQN, the `Bool` ops), or the ability-impl matchers `Intrinsics.numericIntArith`/`compareIntOrdering`/`showIntShow` (matched by their `AbilityImplementation` qualifier); dispatched in `ExpressionCodeGenerator.generateIntrinsic` | bytecode spliced **at each call site**, no method |
| `data` ctor / `handleCases` / `typeMatch` | generated by `DataClassGenerator` | (automatic from `data`) | — |

Anything else body-less → **`"Function not implemented."`** abort. So the two backend paths you author are
**native method** and **intrinsic**, and the decision is mechanical:

> **Native method** if the leaf is non-generic with one fixed JVM descriptor (its call site resolves to exactly one
> method name). **Intrinsic** if it is generic (its descriptor varies per instantiation — a single static method
> can't serve `add` over byte-width vs long-width vs bignum operands), takes/returns *nothing* (`true`/`false`), or
> **is control flow** (`fold` must branch, not push three operands). When unsure, native method is simpler; reach for
> intrinsic only when the descriptor varies or you need a jump. Note an ability-impl method can go *either* way: a
> fixed-descriptor one (`Eq[String]::equals`) is a mangled static method (row above), but a width-varying one (the
> `Numeric[Int]` arithmetic and `Compare[Int]::lessThanOrEqual`) is an inline intrinsic recognised by its ability-impl
> FQN (`Intrinsics.numericIntArith`/`compareIntOrdering`).

Why non-generic ⇒ native method works: the call site invokes `mangledMethodName(fqn, typeArgs)`
(`CommonPatterns`), which for **empty typeArgs + `Qualifier.Default`** is the **bare name** — matching the fixed
`JvmIdentifier("name")` the native method is generated under. A generic leaf gets a `$TypeArg` suffix that no
single method can match, which is exactly why generics go inline. A **native on an ability-impl method** is also
non-generic, but its FQN carries an `AbilityImplementation` qualifier, so `mangledMethodName` gives `m$X$impl$<hash>`,
not the bare name — the backend must therefore emit the method **under that same mangled name** (it passes it into the
native maker), which is the only reason this needs its own path rather than the exact-FQN `implementations` map.

## Backend file inventory (all under `jvm/src/.../jvm/classgen/`)

- `asm/NativeType.scala` — the FQN → `java.lang.*` map (`types`) and the descriptor helpers
  (`javaSignatureName`/`javaInternalName`/`convertToSignatureString`). **Every abstract type that appears in a
  runtime signature must be in `types`** or it gets a bogus nested-class descriptor (`eliot/lang/Bool$Bool`) that
  doesn't exist. Keys use `systemLangType(name)` = `Qualifier.Default` (type FQNs are stripped to Default by the
  time they reach here; `WellKnownTypes` spells them `Qualifier.Type`).
- `processor/Intrinsics.scala` — the `all: Set[ValueFQN]` of inline-emitted FQNs. Adding an FQN here does **two**
  things: routes it to `generateIntrinsic`, and excludes it from the body-less "not implemented" abort
  (`JvmClassGenerator` filters `!Intrinsics.isIntrinsic(k)`).
- `processor/ExpressionCodeGenerator.scala` — `generateIntrinsic` (author your inline case here), and the call
  dispatch (`generateFunctionApplication` → intrinsic vs `generateNormalFunctionCall`). `createExpressionCode`
  emits any sub-expression; interleave it with `methodGenerator.runNative { mv => … }` for raw ASM.
- `processor/NativeImplementation.scala` — the `implementations: Map[ValueFQN, NativeImplementation]` (exact-FQN
  leaves) **and** the `abilityImplementations` table (`(ability, method, dispatchType, JvmIdentifier => NativeImplementation)`
  — natives attached directly to a body-less impl method, keyed by marker because the impl FQN's index isn't static).
  Each entry hand-writes a method via `classGenerator.createMethod(name, paramTypeFQNs, returnTypeFQN).use { mg
  => mg.runNative { mv => … } }` (**`createMethod` appends the `*RETURN`** — leave the result on the stack); an
  `abilityImplementations` maker receives its `name` (the mangled impl-method name), a plain-`implementations` entry
  hardcodes `JvmIdentifier("name")`. Set `override val impure = true` for a leaf that touches the world (I/O) or
  diverges (`forever`); the backend then **asserts the def is declared `private`** (`verifyNativeVisibility`). Pure
  leaves stay `impure = false`.
- `processor/JvmClassGenerator.scala` — the dispatch above (`createModuleMethod`). Read-only for most tasks; the
  leaf is emitted into **the class named by its FQN's module** and only if `used` reached it.
- `asm/CommonPatterns.scala` — `mangledMethodName`, `valueType` (a `GroundValue`'s erased carrier FQN).

## Recipe A — a native-method leaf (fixed signature)

Two flavors: a **plain leaf** keyed by exact FQN, and a **native attached directly to an ability-impl method** (no
wrapper function). Prefer the latter when the leaf *is* the whole meaning of an `implement X[T]::m` — it removes the
one-line `def m = wrapperLeaf(...)` indirection.

**A1 — plain leaf.** Worked example: `readLineInternal(): String`.
1. **Declare** the body-less `private def readLineInternal(): String` in the jvm `.els` (eliot-layers decides the
   file). `private` iff impure.
2. **Map any abstract type** in the signature into `NativeType.types` (e.g. `Bool` → `classOf[java.lang.Boolean]`).
3. **Register** in `NativeImplementation.implementations`: `(systemEffectValueFQN("Console", "readLineInternal"),
   eliot_lang_Console_readLineInternal)`, hardcoding `JvmIdentifier("readLineInternal")` and writing the body (leave
   the result on the stack; `createMethod` appends the `*RETURN`).
4. **Compile-time counterpart** if the checker must evaluate it (see below).

**A2 — native on an ability-impl method.** Worked example: `Eq[String]::equals(a: String, b: String): Bool` →
`String.equals`.
1. **Declare** the instance **body-less** where eliot-layers puts it — colocated with the *type*, in the base
   `stdlib/.../String.els`: `implement Eq[String] { def equals(a: String, b: String): Bool }` (no wrapper `def`). A
   body-less impl is representation-free, so it belongs in the base like `Compare[BigInteger]`; the natives attach
   per-platform. `import`s the ability (`eliot.lang.Eq`) and `Bool`.
2. **Map any abstract type** in the signature (`Bool` → `classOf[java.lang.Boolean]` in `NativeType.types`).
3. **Register** in `NativeImplementation.abilityImplementations` a `(ability, method, dispatchType, maker)` row —
   `("Eq", "equals", "String", eliot_lang_Eq_String_equals)`. The maker takes the mangled method name and writes the
   body:
   ```scala
   private def eliot_lang_Eq_String_equals(methodName: JvmIdentifier): NativeImplementation = new NativeImplementation {
     override def generateMethod(cg: ClassGenerator): CompilerIO[Unit] =
       cg.createMethod(methodName, Seq(systemLangType("String"), systemLangType("String")), systemLangType("Bool")).use { mg =>
         mg.runNative { mv =>
           mv.visitVarInsn(Opcodes.ALOAD, 0); mv.visitVarInsn(Opcodes.ALOAD, 1)
           mv.visitMethodInsn(Opcodes.INVOKEVIRTUAL, "java/lang/String", "equals", "(Ljava/lang/Object;)Z", false)
           mv.visitMethodInsn(Opcodes.INVOKESTATIC, "java/lang/Boolean", "valueOf", "(Z)Ljava/lang/Boolean;", false)
         } } }
   ```
   `JvmClassGenerator.createModuleMethod` marker-recognizes the impl method (`ImplementationMarkerUtils`) and calls the
   maker with `mangledMethodName(vfqn, typeArgs)` per instantiation.
4. **Compile-time counterpart** — usually required, because a *value-level* ability method is surfaced in the checked
   tree, so the checker reduces it too (see below): `StdlibNativesProcessor.abilityImplNativeFor` recognizes the same
   `("Eq","equals","String")` and reduces two concrete literals, stucking on the impl-method FQN for a runtime operand.

## Recipe B — an inline intrinsic (generic / constant / control flow)

Worked example: the `Bool` ops. Represent `Bool` as `java.lang.Boolean` (Recipe A step 2), then:

1. Add the FQNs to `Intrinsics` (a `langBool(name)` helper mirroring `langInt`) and into `all`.
2. Add a case in `generateIntrinsic`. Constants and stack ops are trivial; **control flow emits only the taken
   branch** — for `fold(cond, whenTrue, whenFalse)`:
   ```scala
   val elseL = new Label(); val endL = new Label()
   createExpressionCode(…, arguments(0)) >>                                   // cond → Boolean
   runNative { mv => unboxBool(mv); mv.visitJumpInsn(IFEQ, elseL) } >>
   createExpressionCode(…, arguments(1)) >>                                   // whenTrue arm only
   runNative { mv => mv.visitJumpInsn(GOTO, endL); mv.visitLabel(elseL) } >>
   createExpressionCode(…, arguments(2)) >>                                   // whenFalse arm only
   runNative { _.visitLabel(endL) } >>
   methodGenerator.addCastTo(valueType(expectedResultType))                   // unify the merge type
   ```
   Share `Label`s across the `runNative` blocks; `COMPUTE_FRAMES` computes the stack-map at the merge (both arms
   must leave one value). Emitting arms in the branches is legitimate: Eliot is pure, so a skipped arm has no
   observable effect (and effectful arms are suspended `IO` values anyway).
3. No `NativeImplementation` entry, no descriptor — it never becomes a method.

## The compile-time counterpart (`*NativesProcessor`)

A runtime leaf that the **checker must evaluate at compile time** (a type-level use, a `where` guard bound, an
`auto`/calculated position, a value-level ability method surfaced in the checked tree, or self-sufficiency for the
LSP where jvm is absent) needs a **compile-time reduction** — a Scala `VNative` rule keyed by the same FQN. Two homes:

- `lang/.../monomorphize/processor/SystemNativesProcessor.scala` — compiler intrinsics the compiler *reasons
  about*: `Function`, `Type`, `true`/`false`/`fold`, `Eq[Type]::equals` (type structural equality), `integerLiteral`.
- `stdlib/.../plugin/StdlibNativesProcessor.scala` — library ops the compiler *supplies but doesn't reason about*:
  the `BigInteger` arithmetic backing `Int` bounds, `&&`/`||`/`!`, `Eq[String]::equals`. Add here by default.

Both `Eq[Type]::equals` and `Eq[String]::equals` are the **native-on-impl-method** variant: keyed not by an exact
FQN but by the impl marker (`ImplementationMarkerUtils.isImplementationMethodFor`) — `SystemNativesProcessor.systemContribution`
and `StdlibNativesProcessor.abilityImplNativeFor`. Because `Eq` is multi-instance, bind **only** the impl-method FQN,
never the ability-method `Eq.equals` (a global native there would answer for every instance); and stuck on that same
impl-method FQN so `renormalize` re-fires through the binding the checker fetched. (The fixed exact-FQN natives —
arithmetic, `&&`, `fold` — stuck on their own FQN as before.)

Shape: a curried `VNative` that reduces **only when its arguments are concrete `VConst`**, else returns
`stuck(fqn, args*)` (a `VStuckNative`) so the unifier falls back and `Evaluator.renormalize` re-fires it once the
args concretise. Fail-safe by construction — a leaf reaching the checker with non-concrete args stalls loudly, it
never answers wrong.

The two realizations coexist: `StdlibNativesProcessor`'s `Eq[String]::equals` (checking) + `NativeImplementation`'s
`Eq[String]::equals` (codegen) are different mechanisms on the same impl-method FQN — the merger reads the native for
checking, the backend emits the leaf for codegen, no conflict. For a *runtime* string operand the checker's native
stucks and the residual is a call to the impl method, which the backend realizes. Register a new stdlib label with the
merger via the plugin's `configure()`; the two existing labels are already wired.

**Ability instances across tracks.** An `implement X[T]` needs to *resolve* on whichever pool asks. A jvm runtime
instance is *borrowed* into the compiler pool for resolution, so often no compiler copy is needed. You need a
`stdlib/eliot-compiler/` overlay copy only when the compiler must be **self-sufficient** (LSP type-checks without jvm)
*and* the concrete instance body it would borrow is a jvm-only representation the compiler can't run — that is the
`Either`/`Option` carriers (`data` + folds). But when the instance is **body-less** (its methods are native leaves
attached per-platform), the declaration is already platform-neutral, so it lives **once in the base** and every pool
borrows it — no overlay, no duplication. `Eq[String]` is exactly this: one body-less `implement Eq[String]` in
`stdlib/.../String.els`, the JVM backend attaches `String.equals`, `StdlibNativesProcessor` attaches the compile-time
fold. Compiler-*intrinsic* instances whose leaf has no runtime meaning (types are erased) live in `lang` on both pools
with only the checking-time native — that is `Eq[Type]`.

## Why the runtime was empty — the folding reality (read before "it doesn't run")

The single NbE evaluator forces every **pure** computation at compile time; the monomorphized body that reaches
codegen is the **normal form** (`PostDrainQuoter` reads it back structurally). So a lower-layer pure op **cannot be
observed at runtime unless one operand is a *stuck neutral*** — something the evaluator can't reduce, which bottoms
out at an **effect result** (a native leaf like `readLine`) or a runtime function parameter. This is why `Bool`
had no runtime bytecode for years: every `Bool` traced back to `true`/`false` literals and folded away. **To
exercise your leaf at runtime, drive it from an effect** (`readLine == "yes"`), not from constants.

`PostDrainQuoter` specifics worth knowing:
- Value refs and applications are read back **structurally** — `&&(a,b)` survives to codegen as a
  `MonomorphicValueReference` application. That is *how* your intrinsic case gets reached.
- `fold` is special (`trySelectFold`): it evaluates **only the condition**; a ground `true`/`false` collapses to
  the chosen arm at compile time, an abstract condition leaves an ordinary `fold` application for the backend.
- `trySelectFold` does **not** resolve ability calls, so a *constant* `"a" == "a"` (which reaches its leaf through
  the `Eq` ability) does **not** compile-time-fold in a fold condition — it runs. Direct natives (`&&`) do fold.
  Not a bug; the reduction still fires in resolved-ability positions (guards), same as `typeEquals`.
- A fully-ground compile-time value is *materialised* back to a literal / constructor / `true`/`false` ref
  (`materialise`), which is why a folded `Bool` still needs `true`/`false` handled at codegen.

## Verify

1. `./mill jvm.compile stdlib.compile` — fast Scala check of the backend edits.
2. Build + run a program that **drives the leaf from an effect** so it isn't folded:
   `./mill examples.run jvm exe-jar examples/src/ -m <M>` then `echo <input> | java -jar target/<M>.jar`.
3. **Confirm it's actually there**: `javap -c -p -classpath target/<M>.jar <Class>` — look for your `invokestatic`
   / `GETSTATIC` / `IFEQ`, not a folded constant. `unzip -l target/<M>.jar` shows which classes survived.
4. `./mill jvm.test lang.test stdlib.test` — the jvm integration tests run the whole pipeline; also run an
   Int-arithmetic example (`Ranges`, `ImplicitIntReturn`) since arithmetic shares the intrinsic/`NativeType` path.

## Gotchas

- **Abstract type absent from `NativeType.types`** → bad descriptor / `NoClassDefFoundError`. Map it (to a boxed
  wrapper, never a primitive — carriers are `Object`-erased).
- **Generic leaf as a native method** silently mismatches the mangled call name → "not implemented" or a link
  error. Generics go inline.
- **Forgot `Intrinsics.all`** → your inline FQN hits the body-less abort instead of `generateIntrinsic`.
- **Impure leaf not `private`** → build-time `visibilityViolation`. Set `impure = true` *and* declare it `private`.
- **`createMethod` already appends the return** — leave exactly one value (or none, for `Unit`→`Void`/`ACONST_NULL`)
  on the stack; don't emit your own `RETURN`.
- **FQN qualifier**: `systemLangType`/`langInt`/`langBool` build `Qualifier.Default` FQNs (what reaches the backend
  and `NativeType`); `WellKnownTypes` type FQNs are `Qualifier.Type`. Match Default for backend maps.
- **`.els` imports**: the `eliot.lang` prelude (`ModuleName.defaultSystemModules`: `BigInteger`/`Bool`/`Compare`/
  `Either`/`Eq`/`Function`/`Int`/`Interval`/`Numeric`/`Option`/`Pair`/`Runtime`/`Show`/`String`/`Unit`) + `Type` are
  ambient; every effect — and the jvm carrier `eliot.jvm.IO` — needs an explicit `import`. A missing import shows as
  "Name not defined" (with a cascade onto callers).
- **A leaf is only emitted if `used` reaches it** — an unreferenced native produces no method; test through a real
  call from `main`.
